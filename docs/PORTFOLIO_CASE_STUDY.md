# Solar Site Precheck — Portfolio Case Study

## 1. 何を作ったか

Solar Site Precheckは、太陽光発電候補地をSolar Proへ入力する前に必要な情報をまとめる、社内向けの作業補助Webアプリです。

発電量を直接計算するアプリではなく、候補地選定とSolar Pro入力前の準備作業を短縮することを目的にしています。

主な対象業務は次の通りです。

- 候補地の住所・緯度経度・3次メッシュ確認
- NEDO MONSOLA-11の積雪10cm以上出現率の確認
- 国土地理院DEMを使った地形・地平線の概算
- Solar Proへ読み込める地平線CSVの出力
- 図面PDFのページ選択、回転、注記、PDF/JPG保存
- 法務局資料PDFから相続登記候補を抽出

## 2. なぜこの構成にしたか

最初からSolar Pro本体と直接連携するのではなく、独立したWebアプリとして作りました。

理由は3つあります。

1. Solar Proの公式APIがない可能性が高い
2. 業務では「入力前の確認・整理」に時間がかかっていた
3. 候補地ごとの根拠資料をチームで共有しやすくしたかった

そのため、アプリの役割を「計算ソフトの代替」ではなく「入力準備と判断材料の整理」に限定しています。

## 3. 技術的に難しかった点

### 3.1 3次メッシュとNEDO値の整合性

NEDO MONSOLA-11の積雪・日射データは3次メッシュ単位で扱われます。

候補地座標から3次メッシュコードを計算し、PDFまたはWebから取得したNEDOデータのメッシュと一致する場合だけ値を採用するようにしました。

これにより、見た目には近い地点でも、別メッシュの値を誤って採用するリスクを下げています。

### 3.2 DEMによる地平線の概算

国土地理院DEMを使い、候補地から各方位へ距離ごとの標高を取得して地平線角を概算しています。

単純な標高差だけでなく、距離に応じた地球曲率補正と、保守的な樹高加算を入れています。

ただし、建物・個別樹木・造成後地形までは反映できないため、画面とレポートには「概算」であることを明示しています。

### 3.3 Solar Pro向けCSV出力

Solar Proが読み込めるSunEye形式の `ObstructionElevations.csv` を実ファイルから解析し、アプリの地平線結果を0〜360°の1°間隔に補間して出力するようにしました。

これにより、手入力ではなく、DEM解析結果をSolar Proへ渡せるようになりました。

### 3.4 PDF業務のブラウザ内処理

候補地図面や社内資料を外部サイトへ送らずに扱うため、PDFのページ選択、回転、注記、画像貼り付け、PDF/JPG保存をブラウザ内で処理しています。

回転後の座標と保存時の座標がずれる問題がありましたが、最終的には「保存時にページを回転状態で再レンダリングし、その上に注記を配置する」方式へ整理しました。

## 4. テストと検証

現在はNode標準のテストランナーを使い、次のような純粋関数を中心に回帰テストを入れています。

- 3次メッシュ計算
- 座標入力パーサー
- NEDO積雪値の整合性検証
- 積雪補正係数
- Solar Pro地平線CSVの補間とヘッダー構造
- 実行環境判定

v1.22では、PDFツール画面を分離したうえで、Reactコンポーネントのスモークテストも追加しました。

## 5. レビューを受けて改善した点

このプロジェクトでは、外部レビューやAIレビューを受けながら改善を重ねました。

主な改善は次の通りです。

- App.jsx肥大化への対策として、PDFツール画面を `PdfToolsPage` に分離
- 国土地理院DEMへの一斉アクセスを避けるため、標高取得に同時実行数制限を追加
- Cloudflare PagesとPortable実行環境の差を診断表示で確認できるように整理
- NEDO値や地平線値が「実測値ではなく概算」であることを画面・レポートへ明記
- GitHub Actionsでテストとビルドを自動確認できるようにした

## 6. 現在の課題

ポートフォリオとして見せる場合、まだ改善余地があります。

- `App.jsx` にはPDF以外の大きな状態管理が残っている
- 相続登記チェックもPDF業務ツールとして分離した方がよい
- React Testing LibraryのようなUI操作テストはまだ入っていない
- GitHub ReleasesやCloudflare Pagesへの自動デプロイは整理途中

## 7. 次に進める改善

次の順序で進めると、実務ツールとしてもポートフォリオとしても見せやすくなります。

1. PDF状態・処理を `usePdfTools` フックへ分離
2. 相続登記チェックを `InheritanceCheckPage` として分離
3. 候補地状態を案件JSONとして保存・再読込できるようにする
4. Cloudflare Pagesへの本番デプロイ手順をCI/CD化
5. 主要画面のコンポーネントテストを増やす

## 8. このプロジェクトで示せること

このアプリは、単にAIで画面を作ったものではなく、実務上の課題を見つけ、試作し、現場で検証し、レビューを受けて改善してきたプロジェクトです。

ポートフォリオとしては、次の点を説明できます。

- ドメイン業務を理解してツール化したこと
- 公開データの制約と精度限界をUIに反映したこと
- 実務で使いながら段階的に設計を改善したこと
- テスト、ビルド、配布、更新まで運用を考えたこと
